OVERVIEW

The On-Ramp Problem

Zscaler's security and access controls only work if traffic actually reaches Zscaler first. Connectivity Services is the collection of mechanisms — agents, virtual appliances, network tunnels — that get traffic from wherever it originates to the Zscaler cloud enforcement layer.

There is no single solution that fits every scenario. The right on-ramp depends on where the user is, what device they're on, and what type of access they need. This domain tests your ability to match the right mechanism to the right scenario.

Core principle: Traffic must reach a Zscaler enforcement node before it can be secured, inspected, or policy-controlled. Connectivity Services is how you make that happen — regardless of device type, location, or network environment.
ZSCALER CLIENT CONNECTOR

ZCC — The Endpoint Agent

Zscaler Client Connector (ZCC) is a lightweight agent installed on managed endpoints. It operates at the network layer, intercepting all traffic from the device and forwarding it to the appropriate Zscaler service — ZIA for internet traffic, ZPA for private application access.

Tunnel Modes

Tunnel 1.0
Forwards web traffic only (ports 80/443). Lighter and older mode — does not cover non-web application traffic.
Tunnel 2.0
Forwards all traffic at the network layer, regardless of port or protocol. Required for full ZIA coverage including non-web apps like SSH, custom protocols, and UDP traffic.
No Tunnel / Direct
Used when a trusted network is detected and ZIA is handled by another mechanism (e.g. GRE/IPSec from the network). ZCC steps aside for ZIA but still maintains a ZPA tunnel for private app access.

Trusted Network Detection

ZCC is context-aware. When it detects it is on a known corporate network, it can change its tunneling behaviour. This avoids unnecessarily forwarding traffic through Zscaler when the network itself already provides a forwarding path. Off-network (home, café, airport), ZCC tunnels all traffic to the nearest Zscaler enforcement node.

Split Tunneling

Certain destinations — most notably Microsoft 365 — can be excluded from the ZCC tunnel and allowed to go direct to the internet. This improves performance for latency-sensitive applications like Teams, following Microsoft's own recommendation to avoid backhauling real-time traffic through a proxy.

!
WATCH OUT
Split tunnel improves performance but reduces visibility. Excluded traffic bypasses ZIA inspection entirely. The tradeoff is intentional — but understand that security is the cost of the performance gain.

Device Posture

ZCC continuously collects signals about the endpoint — OS patch level, antivirus status, disk encryption, device enrolment. These signals feed into policy decisions. Split tunnel exclusions and ZPA access can be conditioned on device posture — the device must prove it is healthy before earning the right to go direct or access private apps.

Remember: ZCC's role is traffic interception, tunnel management, and device posture reporting. Authentication itself is handled by the identity provider — ZCC presents the user to Zscaler, it does not authenticate them.
BRANCH CONNECTOR

Branch Connector — ZCC for the Network

Branch Connector solves the problem ZCC cannot: devices that cannot run an agent. Printers, IoT devices, conference room systems, and unmanaged infrastructure cannot install ZCC. Branch Connector is a lightweight VM that acts as the network-level on-ramp to Zscaler, forwarding all branch traffic on behalf of every device connected to it.

Branch Connector vs GRE/IPSec

Feature GRE / IPSec Branch Connector
Setup Manual router/firewall config Zscaler-native VM deployment
User identity context None — source IP only Carries user and device context
ZPA support No Yes — acts as ZPA on-ramp
Policy granularity Location-based only Identity-aware policies
Deployment flexibility On-prem only On-prem, public cloud, data centre

Deployment Environments

  • Physical branch offices — replaces manual GRE/IPSec router configuration
  • Public cloud (AWS, Azure, GCP) — forwards cloud workload traffic through ZIA without ZCC
  • Data centres — covers servers and infrastructure that cannot run agents
Key insight: Branch Connector is not just for branch offices. Despite the name, it is an agentless network on-ramp for any environment where traffic exists but ZCC cannot be deployed.
BROWSER ACCESS

Browser Access — Agentless ZPA

Browser Access enables users to reach internal web applications through their local browser — no ZCC agent required. Zscaler acts as a secure reverse proxy in the cloud: the user's request is authenticated and policy-checked before being proxied through an App Connector to the internal application. The user never touches the internal network directly.

Browser Access Flow

1
User types hr.company.com into their browser
2
CNAME record redirects DNS resolution to Zscaler's infrastructure (configured in the public DNS zone)
3
A record resolves to a ZPA Public Service Edge — not the internal IP
4
Browser connects to Zscaler — no certificate error because a trusted cert is installed for the domain
5
Unauthenticated user is redirected to the IdP for authentication
6
User authenticates — IdP returns a SAML assertion
7
ZPA evaluates the Access Policy — is this user authorised for this app?
8
ZPA proxies the request through an App Connector to the internal application
9
User sees the internal app rendered in their local browser
!
WATCH OUT
Browser Access is HTTP/S only. It cannot handle RDP, SSH, thick-client apps, or custom TCP protocols. For those scenarios, ZCC + ZPA or Privileged Remote Access is required. This is a common exam distractor — don't confuse Browser Access with full ZPA agent access.
!
WATCH OUT
Browser Access ≠ Cloud Browser Isolation. Browser Access renders the app in the user's local browser. Cloud Browser Isolation is a separate feature that runs the browser session remotely. They can be combined, but they are not the same thing.
SD-WAN INTEGRATION

SD-WAN — Network-Level Steering to Zscaler

SD-WAN devices are already steering and managing traffic across the WAN. Integrating with Zscaler means pointing that internet-bound traffic to Zscaler for inspection rather than directly to the internet. The result is local internet breakout with cloud-delivered security at every branch.

Why SD-WAN Alone Is Not Enough

SD-WAN excels at traffic steering, path selection, and bandwidth optimisation — but it has no concept of user identity, device posture, or content inspection. It cannot detect malware, enforce DLP, or apply identity-aware policy. Zscaler provides the security enforcement layer that SD-WAN lacks.

Integration Methods

GRE / IPSec Tunnels
The SD-WAN device forwards internet-bound traffic to Zscaler via a GRE or IPSec tunnel. A Location is configured in ZIA tied to the tunnel source IP. Simple and widely supported, but limited to location-based policy — no user identity context.
Native SD-WAN Integrations
Many SD-WAN vendors (Cisco Viptela, VMware SD-WAN, Versa, Palo Alto Prisma) have built native Zscaler integrations. These automatically establish tunnels, load balance across ZENs, health-check connections, and select the geographically nearest enforcement node — removing manual configuration overhead.

Local Internet Breakout

The old hub-and-spoke model backhauled all branch internet traffic to headquarters, through a central firewall, and back out again. With SD-WAN + Zscaler, each branch breaks out to the internet locally — traffic goes from the branch to the nearest Zscaler enforcement node, gets inspected, and continues to its destination. No hairpin, no HQ bottleneck.

Mental model: Think of Zscaler as replacing the central perimeter firewall — but instead of one firewall at HQ, every branch gets the same security policy enforced at the nearest cloud node. The perimeter moves from a physical location to the cloud.
CLOUD CONNECTOR

Cloud Connector — Zero Trust for Workloads

Traditional cloud networking connects VPCs and VNets via VPN meshes, transit gateways, and peering connections — all workloads end up on shared routable network space. If one workload is compromised, the entire mesh is reachable. Cloud Connector solves this by routing all workload traffic through the Zscaler Zero Trust Exchange instead.

Core idea: Cloud Connector is the agentless on-ramp for cloud workloads — the same role ZCC plays for managed endpoints, but for VMs, containers, and microservices inside VPCs and VNets.

What Cloud Connector Does

Cloud Connector is a virtual appliance that sits inside a VPC/VNet and intercepts all outbound workload traffic — both workload-to-internet and workload-to-workload traffic across VPCs. All traffic is routed through Zscaler for inspection and policy enforcement. No direct VPC peering, no transit gateway meshes, no flat routable networks between workloads.

Where It Deploys

  • AWS, Azure, Google Cloud — public cloud VPCs and VNets
  • Multi-cloud — connecting AWS and Azure workloads without direct peering
  • Hybrid cloud — on-prem data centre to cloud migration paths
  • Can replace ExpressRoute in Azure migration scenarios

Cloud Connector vs Branch Connector

Both are VM-based agentless on-ramps — the distinction is what they serve:

Connector Serves Typical environment
Branch Connector Agentless devices and users (IoT, printers, unmanaged endpoints) Branch offices, physical network segments
Cloud Connector Cloud workloads (VMs, containers, microservices) VPCs, VNets — AWS / Azure / GCP
⟳
ZERO TRUST PRINCIPLE APPLIED
Network adjacency inside a VPC does not imply trust. Cloud Connector enforces the same Zero Trust segmentation for workload traffic that ZPA enforces for user traffic — workloads only reach what they are explicitly authorised to access.
EXAM TIPS

Things to remember

★
MENTAL NOTE
ZCC does not authenticate the user — the IdP does. ZCC's job is to intercept traffic, tunnel it to Zscaler, and report device posture. This distinction trips people up.
★
MENTAL NOTE
ZPA always needs a tunnel from ZCC — even on a trusted network. ZIA might step aside when on trusted network, but ZPA access is identity-driven, not location-driven. The ZPA tunnel stays up regardless.
★
MENTAL NOTE
Split tunnel = performance gain, but security tradeoff. Traffic that bypasses the tunnel bypasses ZIA inspection. Device posture is what earns the right to go direct — it's a privilege, not a default.
★
MENTAL NOTE
Branch Connector works with ZPA, not just ZIA. This is the key differentiator over GRE/IPSec — it can serve as the ZPA on-ramp for agentless devices in the branch.
⟳
ANALOGY — ZCC
ZCC is like a smart traffic controller on your laptop. Depending on where you are, it decides which lanes to open — the ZIA inspection lane, the ZPA private app lane, or both. It knows the roads and routes accordingly.
⟳
ANALOGY — Browser Access
Browser Access is like ordering room service instead of going to a restaurant. You never enter the kitchen (internal network) — someone fetches the food (Zscaler via App Connector) and brings it to you. You get access to what you need without ever being "inside."
⟳
ANALOGY — SD-WAN + Zscaler
SD-WAN is like a smart highway system that routes trucks efficiently. But it has no customs checkpoint. Zscaler is the customs checkpoint — inserted at every exit ramp so every truck gets inspected, regardless of which road it took to get there.
