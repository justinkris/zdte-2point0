QUESTIONS & RATIONALE

All 15 Questions, Answers & Why

Q1 · ARCHITECTURE
CORRECT
A global organisation wants EU users to egress through EU data centres and US users through US Public Service Edges, for compliance. Which Zscaler capability addresses this?
Answer: B — Subclouds
Why: Subclouds let you define which enforcement nodes a tenant is allowed to land on, giving regional data-residency and policy control. SIPA is about source IP identity, Tenant Restrictions is about SaaS tenant scoping, and China Premium Access is a separate regional offering.
Q2 · IDENTITY
CORRECT
In a SAML authentication flow with ZIA, which component acts as the Service Provider (SP)?
Answer: C — The Zscaler Service Edge
Why: The SP is the application that wants to authenticate the user — in this case Zscaler itself. The IdP (Okta, Entra ID, etc.) is the authority that issues the assertion. Browser is just the courier. Remember: SP needs something, IdP provides it.
Q3 · CONNECTIVITY / TROUBLESHOOTING
CORRECT
A user can reach an App Segment from home but not from the office. Same Client Connector version. Most likely cause?
Answer: B — The office network matches a Forwarding Profile that bypasses ZPA for that destination
Why: Forwarding Profiles are network-aware. On trusted/office networks, ZPA can be set to bypass for internal destinations. The giveaway was "same CC version at both locations" — which eliminates agent issues and points at network posture.
Q4 · CONNECTIVITY
CORRECT
Which statement is TRUE about Zscaler Branch Connector?
Answer: C — It forwards traffic from branch locations and server workloads to the ZTE without requiring a Client Connector
Why: Branch Connector is an appliance/VM that sits at a branch or in a workload environment and forwards traffic to the Zero Trust Exchange. It does NOT replace App Connectors (those are for ZPA private-app brokering), and is not installed on end-user devices.
Q5 · ACCESS CONTROL / SSL
CORRECT
Where are "Do Not Inspect" exceptions for SSL traffic to financial sites typically configured?
Answer: B — SSL Inspection policy with a "Do Not Inspect" rule
Why: SSL Inspection policy has explicit allow/inspect/bypass logic, including categories like Finance or Healthcare where privacy law may forbid decryption. URL Filtering and DNS Control operate at different layers and can't control decryption behaviour.
Q6 · CYBERTHREAT / SANDBOX
CORRECT
Which layer of Zscaler's Cloud Sandbox analysis runs the file in an isolated environment to observe runtime behaviour?
Answer: D — Dynamic Analysis
Why: Static analysis reads the file without running it. Dynamic analysis actually detonates the file in a sandbox VM and watches what it does (process creation, C2, file ops). The full pipeline is: Inline AV/YARA → ML → Static → Dynamic → Secondary Static.
Q7 · CYBERTHREAT / QUARANTINE
CORRECT
With AI-driven quarantine enabled, what happens to an unknown executable from a newly observed domain?
Answer: C — File is held at the proxy until analysis completes, then delivered or blocked based on verdict
Why: This is the core of Zscaler's "patient zero" prevention story. Because Zscaler is inline and proxy-based, it can hold a file mid-stream while the sandbox pipeline runs — something out-of-band solutions cannot do.
Q8 · CYBERTHREAT / ITDR
CORRECT
Which Zscaler capability detects attacks against Active Directory (Kerberoasting, DCSync, credential misuse)?
Answer: B — ITDR
Why: ITDR (Identity Threat Detection and Response) is specifically about identity-layer attacks — AD-centric. Deception lures attackers to fake assets. App Protection defends exposed web apps. ATP is broader URL/threat reputation. Watch for this: ITDR ≠ Deception on the exam.
Q9 · DATA PROTECTION
CORRECT
Prevent code uploads to personal GitHub while allowing corporate GitHub. Best enforcement?
Answer: B — Tenant Restrictions + Inline DLP
Why: Tenant Restrictions scopes which GitHub instance a user can reach (personal vs corporate). DLP then inspects what's being uploaded to make sure it isn't source code or secrets. It's the canonical "tenant scope + content inspection" pattern.
Q10 · ZDX
CORRECT
ZDX Score: Device 95 / Network 45 / App 90. Where do you investigate?
Answer: B — The network path (latency, loss, ISP, Wi-Fi)
Why: ZDX decomposes experience into three pillars so you can triangulate. 95 and 90 mean the endpoint and the app are fine; 45 on Network is the smoking gun. This is exactly the scenario-style question you'll see on the exam.
Q11 · ZPA
CORRECT
Which is correct about ZPA App Connectors?
Answer: A — They initiate outbound connections to the ZTE and never require inbound firewall rules
Why: This is the core Zero Trust principle for ZPA: apps are never exposed to the internet because the Connector dials out to the broker. No inbound holes, no DMZ, no VPN. This question is almost guaranteed to appear in some form on the real exam.
Q12 · PLATFORM / SIPA
CORRECT
A SaaS vendor needs to see traffic from a known corporate IP, even though users egress through Zscaler globally. Which feature?
Answer: B — Source IP Anchoring (SIPA)
Why: SIPA re-routes specific traffic so it egresses via a defined corporate IP. Preserves SaaS IP allowlists while still letting traffic benefit from Zscaler inspection upstream.
Q13 · RISK MANAGEMENT
CORRECT
Which is NOT a component of the Risk360 framework?
Answer: D — Incident Response SLA
Why: Risk360 aggregates risk across four breach stages: External Attack Surface → Compromise → Lateral Propagation → Data Loss. "Incident Response SLA" is an ops metric, not a Risk360 stage.
Q14 · AUTOMATION / ONEAPI
CORRECT
To integrate Zscaler with a SOAR for automated quarantine, which capability?
Answer: B — OneAPI
Why: OneAPI is the unified API fabric across ZIA, ZPA, ZDX, and Zidentity — consistent auth, consistent access model. LSS is for log streaming, not action; Nanolog is an internal log tier; Browser Access is agentless ZPA. For any "SOAR / workflow / automation" question, think OneAPI.
Q15 · ORDER OF EXECUTION
CORRECT — C & D
SSL Inspection occurs BEFORE which of the following policies so decrypted content can be evaluated?
Answer: C (URL Filtering & DLP) AND D (Tenant Restrictions)
Why: URL Filtering needs the full path and DLP needs the request body — both require decryption. Tenant Restrictions also requires SSL inspection because it injects the Restrict-Access-To-Tenants header into the HTTPS request, which is only possible once the session is decrypted.
KEY INSIGHTS

Big takeaways from this mock

1. Inline proxy architecture is the "why" behind most answers

Notice how many correct answers (Q6, Q7, Q11, Q15) trace back to the same architectural fact: Zscaler sits inline as a proxy. That's why it can quarantine files mid-transfer, inject headers, broker without opening firewalls, and inspect decrypted traffic. When a question feels ambiguous, default to reasoning from the inline-proxy model.

★
CORE PATTERN
"Zscaler is inline and proxy-based" → can hold, modify, inject, decrypt, log. Out-of-band tools can't. Many ZDTE questions are just this idea in disguise.

2. Tenant Restrictions requires SSL inspection

The mechanism isn't SNI-based tenant detection — it's header injection. Zscaler inserts Restrict-Access-To-Tenants into the HTTPS request so Microsoft evaluates which tenant the user is allowed to sign into. Header injection is impossible without decryption.

!
CLARIFIED FROM SESSION
Earlier I suggested Tenant Restrictions could work off SNI alone — that was wrong. The documentation is explicit: Tenant Restrictions requires SSL inspection to inject the tenant-scope header. If the session isn't decrypted, Tenant Restrictions can't function.

3. ZDX's three-pillar score is a triage tool, not a single number

Q10 is the template ZDX scenario question: the gap between the three scores tells you where to look. Always read Device / Network / App as independent signals.

ZDX Pillar What It Measures If Low, Investigate…
Device CPU, memory, battery, Wi-Fi strength The endpoint itself
Network Latency, packet loss, jitter on the path LAN, ISP, VPN, hop-by-hop
Application App response time, HTTP errors SaaS / app back-end
EXAM TRAPS

Distractor patterns to watch

→
TRAP — ITDR vs DECEPTION
Both are under Cyberthreat. ITDR = identity attacks (AD, Kerberoasting, DCSync). Deception = fake lures (honeypots, fake creds, fake servers). They sound similar; the exam will place them as distractors for each other.
→
TRAP — BRANCH vs CLIENT vs APP CONNECTOR
Client Connector — endpoint agent on user devices.
Branch Connector — appliance/VM at a branch or for server workloads; no CC needed.
App Connector — ZPA-only, brokers private-app access, outbound-only.
Never let the exam confuse these.
→
TRAP — SAML ROLES
In a ZIA/ZPA SAML flow, Zscaler = Service Provider (SP), IdP = Okta / Entra / Ping. The browser is just the messenger, not a role. Many candidates pick "the IdP" as SP because it's the most prominent name.
→
TRAP — SUBCLOUDS vs SIPA
Subclouds — which nodes a tenant lands on (regional/residency control).
SIPA — the source IP a SaaS sees (identity preservation for allowlists).
Different layers, often confused on exam questions about "control where traffic goes."
⟳
ANALOGY — TENANT RESTRICTIONS
Think of Tenant Restrictions like a bouncer slipping a note into your application form: "this person may only enter the Corporate ballroom, not the Personal one." To slip that note in, the bouncer has to open the envelope first (decrypt SSL). No decryption, no note, no restriction.
